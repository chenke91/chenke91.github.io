# Pi 与自修改软件：AI Agent 时代的工程质量、工具边界和复杂度控制

## 简介

本文整理自 The Pragmatic Engineer 的访谈视频 [Building Pi, and what makes self-modifying software so fascinating](https://www.youtube.com/watch?v=n5f51gtuGHE)。访谈对象是 Pi 作者 Mario Zechner，以及 Flask 作者、Pi 早期使用者与贡献者 Armin Ronacher。核心主题包括：为什么一个极简 coding agent 可以通过“自修改”获得强扩展性，AI agent 如何改变工程团队协作，为什么软件质量会因 agent 产能提升而下降，以及 MCP 与 CLI 在 agent 工具体系中的取舍。

---

## 核心要点

- **Pi 的核心价值不是功能多，而是可被自身修改。** 它只保留 read/write/edit/bash 等极少工具和稳定 agent loop，通过 TypeScript 扩展点让用户或 agent 自己把缺失能力补进去，例如 MCP、计划模式、UI 定制、调试工具等。
- **AI agent 放大了代码产出，也同步放大了错误、复杂度和评审压力。** 即使 agent 错误率低于人类，只要产出速度高很多，代码库退化速度仍会提升；人类 reviewer 无法按同等速度消化 3k～10k 行/天的机器生成代码。
- **复杂度是人和 agent 共同的敌人，但 agent 更容易制造复杂度。** agent 不会“感到痛苦”，不会因未来维护成本主动重构；它常通过兜底、恢复、配置分支等方式制造更多状态空间，最终让自己也无法在上下文窗口中理解代码库。
- **非工程角色参与软件生产是新现实，但必须有流程护栏。** PM、设计、营销、销售可以用 agent 做原型、demo 或 PR，这能提升参与度；但直接进入主代码库会制造集成、责任、质量和产品真实性风险。
- **未来的 agent 工作流需要“可验证环境”，而不是只追求自主和并行。** 最好的 agent 使用场景是：人仍在 loop 中，系统能自动验证变更，agent 能根据截图、状态 dump、测试结果等反馈自我修正。
- **MCP 更像把工具结果塞进上下文，CLI/代码执行更擅长组合。** MCP 在消费级连接外部服务、企业认证场景有价值，但大量 openAPI-to-MCP 服务器会导致工具爆炸、上下文膨胀和不可组合；CLI 管道和代码执行让模型只看最终结果，更适合复杂组合任务。
- **“慢下来”不是反 AI，而是把 AI 用在减痛和提质上。** 与其建一支 dark factory 式 agent 军队生成更多平庸代码，不如先用 agent 自动化组织中令人烦躁但可验证的工作，把节省出的时间用于思考产品、架构和打磨质量。

## 正文

### 1. 背景：从 AI 怀疑者到 coding agent 使用者

Mario 和 Armin 都是资深工程师，对早期 GitHub Copilot 并不乐观。Copilot 的 autocomplete 体验、代码记忆和许可证争议让他们更多把它看作一个有趣但不可靠的工具。真正的转折出现在 coding agent 具备以下能力之后：

- 能进行 tool/function calling；
- 能直接读写文件系统，而不是依赖向量索引；
- 能通过 shell、测试、截图、状态 dump 等方式形成反馈回路；
- 能在较长任务中自主搜索代码、修改代码、运行验证。

Mario 认为 Claude Code 让“agentic search”进入可用阶段：与其先索引代码库，不如直接给 agent 文件系统访问能力，让它自己像工程师一样读文件、找入口、运行命令。Armin 也认为，直到 Claude Code 这类工具出现，AI 才真正让他感觉软件工程工作流会被改变。

### 2. Pi 的设计动机：稳定、极简、透明

Mario 最初是 Claude Code 的重度用户，但在 2025 年夏天开始不满：工具团队为了 dogfood 和高速迭代不断加入功能、修改 system prompt、注入不可见上下文和 reminder，导致原本稳定的工作流突然失效。对他来说，开发工具应该像锤子一样稳定：LLM 本身可以是非确定性的，但 harness 中的确定性部分必须可靠。

他也尝试过 AMP、Droid、OpenCode 等替代品，但不喜欢其中一些“背后替用户处理上下文”的设计，例如：

- 自动裁剪 tool result；
- 每次 edit 后自动把 LSP 诊断塞给模型；
- 插件系统不够开放，想改行为必须 fork；
- 工具自己做了很多模型看不清、用户也难控制的事。

Pi 因此被设计成一个极简 coding agent：

| 设计点 | 含义 |
|---|---|
| 极少内置工具 | 主要是 read、write、edit、bash，不试图内置所有能力 |
| 自有 LLM provider 抽象 | 不直接依赖不符合作者审美的通用 SDK |
| 通用 agent loop | 支持 tool calling、streaming、TUI 等基本能力 |
| 扩展点丰富 | 用户可通过 TypeScript 模块接入自定义工具、压缩策略、TUI 改造等 |
| 同进程加载扩展 | 扩展能直接影响 agent 运行环境，降低自修改门槛 |

它的关键不是“功能最多”，而是“足够容易被修改”。Pi 没有 MCP，用户可以让 Pi 给自己加 MCP；没有计划模式，用户可以让 Pi 写自己的计划模式；TUI 不喜欢，也可以让 Pi 修改 TUI。

### 3. 自修改软件：让工具随任务形态变化

访谈中最有启发的观点是：**未来的软件可能不再是固定工具，而是能根据用户需求自我塑形的 malleable software**。Pi 是这种方向在 coding agent 领域的一个早期实验。

Armin 的例子是用 Pi 做游戏开发。他不是直接让 agent 写游戏，而是先让 agent 帮自己搭建一套适合该项目的验证环境：

- 游戏运行后自动截图；
- 能 dump 状态并让 agent 读取；
- TUI 中可以快速浏览截图；
- 会话可以回退、分叉，便于探索不同实现；
- 针对当前项目构建专属调试和反馈工具。

这说明 agent harness 不应该总是通用形态。不同任务需要不同的感知、验证和交互方式：网页项目、游戏项目、后端服务、数据分析、非技术工作流，可能都需要不同的 agent UI、工具集和反馈回路。

Mario 用施工现场作比喻：工程上不会只用锤子解决所有问题，agent 也不应只有一个通用工具。Pi 的目标不是替所有任务内置最佳流程，而是让 agent 能为具体任务长出合适的工具。

### 4. Agent 产能带来的质量问题

Armin 访谈了 30 多个工程团队，观察到 AI coding tools 的采用往往在假期或空档爆发：人们有 2～3 周时间沉浸式试用后，才真正“开窍”。但大规模采用之后，团队很快遇到质量与流程问题：

- PR 更大、更频繁，review 变得心理负担更重；
- agent 写出的代码经常不像有经验工程师会提交的代码；
- 人类因 automation bias 放松审查，错过糟糕实现；
- junior、PM、营销、销售等非工程角色也能生成 PR 或 demo，增加集成压力；
- agent 生成的“看似完成”功能可能不是真实产品能力，例如销售 demo 做出了不存在的功能。

一个关键差异是：**人类工程师会感觉痛苦，agent 不会。**

人类在维护糟糕接口、复杂状态机、技术债时会感到痛苦，这种痛苦会推动重构、删除功能、拒绝需求和控制复杂度。优秀 senior engineer 的价值往往不是“写得更快”，而是有足够 battle scars，知道什么时候说“不”。而 agent 不会为未来维护成本痛苦，只会继续加代码。

Mario 用一个简单算术描述风险：如果 agent 每天能产出 10 倍代码，即使错误率只有人类一半，也会带来 5 倍错误量；当组织再叠加几十、上百个 agent，review、责任和质量控制都无法线性扩展。

### 5. “Prompt Request”与糟糕实现的价值

面对非工程角色或外部贡献者用 agent 生成的 PR，Peter Steinberger 提出过“prompt request”思路：与其给工程师一个机器生成 PR，不如给 prompt，让工程师理解意图后自己生成或实现。

访谈中对这个观点有两种补充：

- Armin 认为，生成过程本身能帮助提出者澄清“到底想要什么”。但最终代码往往不该直接合并，因为有经验工程师可能会采用完全不同实现。
- Mario 认为，糟糕的 agent PR 仍有价值，因为它展示了“天真实现”会长什么样，等于别人替你探索了一条低质量路径，节省了试错时间。

因此，agent 生成内容不一定应该被当作可合并代码，更适合被当作：

- 需求澄清材料；
- 原型和 demo；
- 反例实现；
- 设计讨论输入；
- 帮助工程师理解用户真正想要什么的“意图载体”。

### 6. 开源维护的新瓶颈：不是代码，而是意图和背压

Pi 和 OpenClaw 的流行让 Mario 直接遭遇了 agent 生成 issue/PR 洪水。OpenClaw 用户的 agent 会自动向 Pi 提 issue 或 PR，有时用户本人甚至不知道。传统开源中，PR 之所以特殊，是因为它需要人投入时间；现在生成 PR 几乎没有成本，导致维护者要处理大量“看似有贡献、实际缺少意图”的输入。

Mario 的应对方式是制造一个“人类意图瓶颈”：

1. 如果贡献者账号不在信任列表中，PR 自动关闭；
2. bot 留言要求对方用“人类语气”开一个不超过一屏的 issue；
3. 如果维护者认可并回复 `looks good to me`，账号进入信任文件；
4. 下次该账号的 PR 才能通过。

这个过滤器有效的原因很有意思：agent 往往看不到 GitHub workflow 留下的评论，因此不会按要求补充 human voice issue。这里的目标不是证明“是不是人类”，而是建立一个维护者可处理的背压机制。

Armin 对开源未来的判断更偏结构性：开源的长期价值仍来自人类围绕难题投入精力、建立社区、维护生态。AI 会制造更多两天就死的项目，但真正能长期维护、形成社区的项目数量未必会爆炸式增长。

### 7. 复杂度：agent 自己制造的上下文敌人

Mario 说，他最大的敌人是复杂度，agent 最大的敌人也是复杂度。

原因很直接：如果代码库有 60 万行，而 agent 上下文窗口只能有效处理约 20 万 token，那么 agent 是否能完成任务，取决于能否找到所有相关上下文。现实中信息检索并未彻底解决，agentic search 也不保证能找到全部相关代码。更糟的是，agent 自己生成的大量代码会继续膨胀代码库，让之后的 agent 更难读懂它。

访谈中提到的典型退化模式包括：

- 不该恢复的错误被 catch 后走默认配置，状态空间变大；
- 为避免失败添加过多 fallback，系统行为更不可预测；
- 每个 feature 都“顺手”加一点配置、状态、适配层；
- agent 为了当前任务局部成功，引入未来全局维护成本；
- 人类不再亲自进入代码细节，失去对复杂度的痛感。

Mario 在 Pi 中的做法是区分代码重要性：HTML export 这种边缘功能可以允许一定 slop，只要输出看起来对；但 agent loop、扩展加载机制等核心路径必须保持高质量。他维护质量的方式是**无情重构**：通过重构迫使自己重新进入代码库、理解结构、降低复杂度。

### 8. 有益摩擦：不是所有 friction 都该删除

AI 工具常追求“无摩擦”：让 agent 更自主、更并行、更少被流程阻挡。但 Armin 提醒，工程组织里有些 friction 是故意设计的安全机制，不是糟糕开发体验。

例如：

- 关键服务配置变更需要更多审批；
- 高优先级服务有 SLO、owner、on-call 和成熟度要求；
- 危险操作需要二次确认；
- 数据库 migration、权限变更、生产发布需要 checklist；
- 新人修改关键系统前需要找有经验的人 socialization。

这些摩擦的意义不是拖慢所有事，而是在真正高风险的地方强迫人重新思考：是否真的要做？是否理解后果？是否有足够上下文和背书？

agent 并行运行的诱惑会让团队想消除所有 friction，因为 agent 本身速度慢，唯一明显省时方式是并行。但如果连有益摩擦也被消除，组织就会失去质量、责任和安全的背压。

### 9. 为什么需要“慢下来”

Mario 的 blog post 主题是 “we all need to slow the f down”。这不是否定 agent，而是反对把 agent 产能直接转化为更多代码、更多功能、更多并行任务。

他的建议是换一个使用 AI 的顺序：

1. 先找出组织中最烦人、最重复、最可验证的工作；
2. 用 agent 自动化这些低价值痛点，让人获得更多时间；
3. 人把节省出的时间用于思考：到底要构建什么？用户真正需要什么？
4. 一旦决定构建，再让 agent 参与实现、验证和打磨；
5. 不要把“生成更多代码”当作首要 KPI。

核心判断是：agent 学到的知识主要来自互联网，而互联网代码质量从垃圾到平庸居多。规格如果不完整，agent 会用训练数据中的平均模式填空。最好的规格其实是软件本身；只要规格不是完整软件，就会有大量空白需要填补，而 agent 填补空白的默认材料未必高质量。

### 10. MCP 与 CLI：上下文工具 vs 可组合执行

访谈最后讨论 MCP 与 CLI。两位嘉宾并不完全否定 MCP，但更偏好 CLI/代码执行。

MCP 的合理场景：

- 消费级聊天应用连接邮箱、OneDrive、日历等外部服务；
- 企业场景中的认证、授权、审计、受控数据访问；
- 小而清晰、专门为某个任务设计的工具；
- 需要把外部服务包装为 agent 可调用能力的场景。

MCP 的问题：

- spec 和实现复杂；
- 很多企业直接把完整 OpenAPI spec 映射成大量 MCP tools，导致工具爆炸；
- 工具输出必须进入上下文，容易快速填满 context；
- 多个 MCP 工具的结果组合需要模型在上下文中做数据转换，可靠性差；
- MCP 抽象会限制 agent 在 shell、文件、大输出处理上的创造性。

CLI/代码执行的优势在于可组合：模型可以写脚本、调用多个命令、用 pipe 或程序处理数据，只把最终结果读回上下文。对于复杂任务，“让模型写代码来组合工具”往往比“让模型在自然语言上下文中组合多个工具结果”更可靠。

一个可能的未来方向是：MCP 负责认证与受控访问，真正组合仍通过 SDK、HTTP API、代码执行或更接近 CLI 的机制完成。

### 11. 对 2027 的谨慎预测

两位嘉宾都避免做确定预测，但给出几个方向：

- 自修改软件会扩展到更多工具，甚至非技术知识工作；
- code execution / code generation 会继续是 agentic task 的核心范式，因为训练数据丰富，也天然能控制计算机；
- 企业需要解决 code generation 与认证、权限、合规、审计之间的矛盾；
- 工程团队会越来越依赖少数模型供应商，甚至出现“没有机器就维护不了代码库”的依赖风险；
- 顶级模型访问权可能分层，只有部分合作伙伴能用到最强模型，形成新的竞争差异；
- 当前的 hype、dark factory、software is dead 叙事大概率会自我修正，因为复杂度和质量问题会在几个月后显现。

## 可复用方法 / 行动清单

### 面向团队的 AI coding agent 使用清单

- **先定义任务风险等级**：边缘功能、原型、内部脚本可以更放手；核心路径、权限、安全、数据、计费、agent loop 等必须强 review。
- **给 agent 建验证环境**：测试、截图、状态 dump、日志、lint、类型检查、端到端验证，比多写 prompt 更重要。
- **保留人类痛感**：定期亲自读核心代码、做重构、删除复杂度，不要只看 agent 输出摘要。
- **限制 PR 尺寸和频率**：机器生成代码也要遵守 reviewer 可处理的上限；大 PR 应拆分为可验证小变更。
- **把非工程贡献转为意图输入**：PM/销售/设计生成的实现可当原型或需求说明，不应默认进入主分支。
- **建立背压机制**：对外部 agent PR/issue 设置人类意图门槛，例如先开简短 issue、说明动机和预期行为。
- **保留有益摩擦**：高风险服务继续使用审批、checklist、SLO、owner、二次确认等机制。
- **把 AI 优先用于减痛**：先自动化重复、烦人、可验证工作，再追求新功能扩张。

### 判断一个 agent harness 是否适合长期使用

| 判断项 | 好信号 | 风险信号 |
|---|---|---|
| 稳定性 | 确定性部分行为可预期 | 频繁注入不可见上下文、系统提示和 reminder |
| 透明度 | 用户能看到工具调用、上下文和状态 | 工具在背后替模型剪裁、改写或隐式诊断 |
| 扩展性 | 能为项目构建专属工具和反馈回路 | 想改行为只能 fork 或等待官方功能 |
| 验证能力 | 易接入测试、截图、日志、状态 dump | 主要靠模型自信地说完成 |
| 复杂度控制 | 鼓励小工具、小核心、项目级定制 | 内置越来越多通用功能和隐式魔法 |

## 关键概念

- **Pi**：Mario Zechner 开发的极简 coding agent，强调稳定核心和高可扩展性，允许 agent 通过扩展点修改自身能力。
- **Self-modifying software / malleable software**：软件能根据用户需求和具体任务修改自身结构、UI、工具和工作流，而不是固定功能集合。
- **Agentic search**：agent 通过文件系统、shell、grep、阅读代码等方式自主搜索相关上下文，而不是完全依赖预先构建的索引。
- **Automation bias**：人看到 agent 某次输出很好后，容易过度相信后续输出，降低审查强度。
- **Prompt request**：把“想做什么”的 prompt 或意图交给工程师，而不是直接提交 agent 生成的 PR。
- **Back pressure / 背压**：为维护者或工程团队设置处理速度上限，防止 agent 生成输入洪水压垮 review、triage 和质量控制。
- **Dark factory**：让大量 agent 并行生成代码、测试、PR 的幻想式自动化工厂；风险是产出很多“某种东西”，但质量、责任和维护性不可控。
- **有益摩擦**：故意设计的流程阻力，用于让高风险变更变慢、可审计、可反思，而不是糟糕开发体验。
- **MCP**：Model Context Protocol，用于把外部服务和工具暴露给模型；适合认证和服务连接，但在大量工具组合和上下文管理上存在限制。
- **CLI/代码执行范式**：让模型通过 shell、脚本、pipe、SDK/API 等方式组合工具，只把处理后的结果放回上下文。

## 我的理解

这期访谈的价值在于，它把 AI 编程从“能不能生成代码”推进到“怎样不让组织和代码库被生成速度反噬”。Pi 的自修改思想很重要：未来 agent 工具可能不是一套通用 UI + 通用工具，而是能围绕项目验证方式、人的交互方式和风险等级动态生成专属 harness。但更重要的是质量观：AI 真正的收益不应只是更多代码，而应是把人从低价值痛点中释放出来，让人有时间做判断、设计、重构和取舍。对于知识库而言，这篇可以和 Harness Engineer、Agentic Engineering、Claude Code 技巧等笔记一起组成“AI 编程工程化”主题脉络。
