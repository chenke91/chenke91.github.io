# Tasklet 的 Andrew Lee：三类软件将存活，AI 原生平台如何重构知识工作

## 简介

本文整理自 YouTube 访谈 [Three Kinds of Software Survive: Tasklet's Andrew Lee on Competing to be a Horizontal Platform](https://www.youtube.com/watch?v=nsWdSaKVbIY&t=4s)。Andrew Lee 讨论了 Tasklet 在 6 个月内几乎重写全部系统后的产品与架构变化，以及他对 AI 原生软件未来形态的判断：未来会只剩少数横向平台、少数头部 API/Headless 产品，以及大量“卖结果”的解决方案公司。

这期内容最有价值的地方，不是某个单点技巧，而是它把**上下文管理、文件系统记忆、计算机使用、组织共享脑、可验证性、权限控制、生成式 UI**这些模块串成了一个完整的 AI 原生工作台蓝图。

---

## 核心要点

- Tasklet 已从“工作流自动化工具”重构为“可同步对话的通用 Agent 平台”，产品逻辑不再围绕单次任务，而是围绕一条持续的长对话和跨任务上下文。
- 他们把“聊天历史”从 LLM 输入中移出，改成“文件系统里保存历史 + LLM 只拿提示和索引”，再通过分层摘要和桶状压缩维护有限上下文，核心目标是降低成本并保留可检索性。
- Computer use 已成为产品核心：shell、文件系统、数据库、浏览器 VM 都是主路径能力，而不是附加功能。
- Andrew 认为未来大多数软件会被压缩为三类：**横向平台**、**Headless/API 产品**、**解决方案公司**；传统带 UI 的 SaaS 竞争空间会显著收缩。
- “Always bet on the models” 仍然成立，但真正的竞争重点正在从模型能力转向 harness：记忆、权限、回滚、审计、生成式 UI、可验证执行链。
- 企业级采用 AI 的关键不是让模型更“聪明”，而是让它在出错时可回滚、可审批、可审计、可测试。
- Tasklet 正在走向“shared brain”路线：组织/工作区/Agent 三层上下文共享，连接、技能、记忆、文件系统都要逐步共享化。

## 正文

### 1. 过去 6 个月：几乎把产品全重写了

Andrew 说，Tasklet 在过去半年里“基本上重写了所有东西”。最重要的变化包括：

- **产品定位变化**：早期是工作流自动化；现在变成可同步对话的通用 Agent。
- **交互形态变化**：用户不再只是在配置 workflow，而是希望“一个大而连续的聊天”里完成日常工作和自动化。
- **技术架构变化**：围绕上下文、计算机使用、集成方式、压缩策略全部重写。

这意味着，Tasklet 不再是“一个会跑任务的工具”，而是在做一个“能承载知识工作全流程的 Agent 工作台”。

### 2. 上下文管理：把历史放进文件系统，把 LLM 变成检索与执行层

这是整期最核心的工程思想之一。

Andrew 明确说，**历史不再是直接喂给模型的主输入**，而是被放入文件系统中；LLM 看到的是：

- 当前需要完成什么任务的提示
- 哪些文件值得读
- 哪些历史需要恢复
- 哪些旧内容可以被更低保真地压缩

他的做法是**分层、桶状、缓存感知**的：

- 最近的对话保留更高保真度
- 更早的内容逐渐丢弃 thinking blocks、tool call responses、tool arguments
- 再往前就进入摘要压缩
- 压缩不是一次性完成，而是随着历史增长逐步更新

这套方法的核心假设是：

1. 最近内容最重要。
2. Agent 应该学会自己去文件系统里查找信息。
3. 通过缓存和分层压缩，可以让长会话成本可控。

### 3. 缓存的角色变了：不是“少用上下文”，而是“让上下文可持续维护”

Andrew 强调，缓存并没有因为文件系统方案而不重要，反而更重要了。

原因是：

- Agent 现在需要更多工具调用去读文件、查状态、恢复历史
- 如果没有缓存，运行成本会非常高
- 他提到 Anthropic 侧使用的是**5 分钟缓存**，适合活跃会话或单次运行
- 不同 provider 的缓存能力不同，OpenAI 的缓存原语更友好

结论是：

- **缓存用于短期减少重复 token 成本**
- **摘要与文件系统用于长期保存可恢复上下文**

这是两个层次的问题，不是同一个问题的替代。

### 4. Computer use 成为核心，而不是附属能力

Tasklet 的另一个大改造，是把 computer use 从“附加功能”变成核心路径。

他们现在的 Agent 会频繁：

- 跑 shell 命令
- 读写文件系统
- 操作数据库
- 使用 headless VM
- 使用浏览器 VM
- 通过浏览器和系统状态跨轮次保持持久性

这说明他们的产品不再只是“文本对话”，而是**可执行环境 + 记忆系统 + UI 生成能力**的组合。

### 5. 集成系统也被重写：连接不只是接上去，而是要可管理、可复用

Andrew 提到，他们重构了连接系统，使 Agent 对外部系统的控制更强。

一个很具体的变化是：

- 以前同类型连接往往只能接一个
- 现在可以连接多个同类型实例，例如多个 Gmail 账号

这类设计指向的是：

- 连接不是一次性配置，而是可共享的基础设施
- 连接应该服务于团队、组织、工作区，而不只是某个单独 Agent

### 6. “Always bet on the models” 仍然成立，但竞争正在转向 harness

Andrew 认为模型能力的提升持续证明了“永远押注模型”的策略是对的。

但他也强调，今天的关键竞争不是“谁有更会聊天的模型”，而是：

- 谁能把模型包进更好的 harness
- 谁能让模型在现实世界里安全执行
- 谁能管理上下文、权限、审计和回滚
- 谁能把模型变成可靠的工作系统

换句话说：

> 模型是大脑，harness 才是能否落地的脊柱和外骨骼。

### 7. 未来软件只剩三类

这是他最鲜明的产业判断。

#### A. 横向平台

例如面向知识工作的通用 Agent 平台，用户希望把很多软件都收进一个系统里。

特征：

- 统一上下文
- 统一连接
- 能做分析、能生成 UI、能执行代码
- 会替代大量日常 SaaS 操作

#### B. Headless / API 产品

例如 Stripe 这类产品。

特征：

- 仍然很重要
- 但 UI 价值下降
- 用户可能只需要 API，不再需要仪表盘

#### C. 解决方案公司

例如律师、房产中介等。

特征：

- 对外卖的是结果，不是软件
- AI 深度参与内部流程，但对用户来说不一定可见

这意味着传统 SaaS 的中间地带会越来越难生存。

### 8. Salesforce 为什么会受冲击

Andrew 认为像 Salesforce 这样的系统会面临更大压力，因为：

- Agents 让系统间迁移更容易
- 很多老代码和复杂 schema 会逐渐变成历史包袱
- 以前依赖“系统很复杂，所以很难替代”的护城河会变弱

但他也没有说 Salesforce 会消失，而是认为：

- 它的价值可能会缩小
- 它的用户界面和冗余功能会越来越不重要
- 更小、更专门的替代产品会出现得更多

### 9. 可靠性与可回滚：AI 平台要像生产系统一样设计

他特别强调，企业不会因为 AI 很强就接受“不可控”。

所以未来平台必须提供：

- **版本化 / 回滚**：Agent 做错时可退回
- **日志与审计**：知道做了什么
- **权限控制**：并非所有动作都自动执行
- **审批流程**：高风险动作需要人确认
- **测试环境**：先验证，再上线
- **生成脚本而不是直接改数据**：尤其是迁移场景

一个很好的例子是数据迁移：

- 差的方式：让模型直接读写数据
- 好的方式：让模型生成迁移脚本、测试、说明，再交给人审批

### 10. Shared brain：组织、工作区、Agent 三层上下文

Tasklet 的下一步，是把上下文从“单个 Agent”提升到更高层级：

- **组织层**：公司使命、价值观、基本规则
- **工作区层**：团队目标、品牌语气、共享资源、技能、OKR
- **Agent 层**：具体任务、具体文件、具体会话指令

目前他们已经做到了部分共享连接，未来会加入：

- 共享技能
- 跨 Agent 记忆
- 共享文件系统

这就是他说的“shared brain”方向。

### 11. Instant apps：生成式 UI 让“任何分析界面”都能一键生成

这是他们最近很成功的功能之一。

用户可以用一个 prompt 生成：

- 数据探索器
- 仪表盘
- 带筛选器和控制项的分析界面
- 挂接在已有连接数据上的专用 UI

这类功能的价值在于：

- 不再需要先决定做什么界面
- 让 UI 变成按需生成的产物
- 直接把 Agent 从“回答问题”推进到“生成工作界面”

### 12. 现阶段的经济结构：token 成本已经是实打实的运营成本

Andrew 提到，Tasklet 内部的 token 支出已经很可观，内部 token spend 大约是 payroll 的 **5% 到 10%**。

这说明：

- AI 不是“便宜的玩具”了
- harness 和缓存优化直接关系到公司财务结构
- 成本管理会越来越重要

### 13. 供应商与实践细节

他还提到几个比较实际的供应商选择：

- **Blackel**：沙箱供应商，冷启动快、性能好
- **Firecrawl**：爬虫/网页抓取
- 他们暂时倾向于自建核心存储与数据库/文件系统能力

也就是说，越接近核心的基础设施，越倾向自己掌控。

## 可复用方法 / 行动清单

### 给做 AI 产品的人

1. 不要只盯模型能力，优先设计 harness。
2. 把聊天历史和长期记忆拆出去，改成文件系统可读的状态。
3. 用分层摘要处理长上下文，不要强行塞满 context window。
4. 让 Agent 先生成脚本、计划、测试，再执行高风险动作。
5. 给关键动作加审批、日志、回滚。
6. 把连接、技能、文件、记忆设计成可共享资源，而不是单 Agent 私有配置。

### 给做企业软件的人

1. 重新评估 UI 的长期价值：很多界面会被 Agent 生成或替代。
2. 区分“头部 API 服务”和“真正需要 UI 的业务”。
3. 关注数据迁移、权限、审计、容错，这些是 AI 落地门槛。
4. 以“结果交付”而不是“功能堆叠”来思考产品形态。

## 关键概念

- **Harness**：包裹模型、约束模型、让模型可执行的系统层。
- **File system as memory**：把长期上下文存入文件系统，而不是塞进 prompt。
- **Bucketed compaction**：按时间分桶、分层压缩历史，减少缓存破坏。
- **Shared brain**：组织/团队/Agent 共享上下文、技能、连接和记忆。
- **Instant apps**：基于连接数据即时生成可交互 UI。
- **Headless software**：软件的价值转向 API，UI 变弱。

## 我的理解

这期最重要的启发是：AI 软件的竞争焦点已经不在“谁能让模型回答得更像人”，而在“谁能把模型变成一个可回滚、可审批、可共享、可扩展的工作系统”。

如果说过去的软件竞争是 UI 和功能，那么接下来更像是**上下文工程 + 生产系统工程 + 组织记忆工程**的竞争。
