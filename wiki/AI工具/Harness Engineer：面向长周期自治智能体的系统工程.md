# Harness Engineer：面向长周期自治智能体的系统工程

## 简介

本文整理自 AI Jason 的视频 [wtf is Harness Engineer & why is it important](https://youtu.be/kJPvfoLtFFY)。视频讨论的不是“怎样写一个更好的 prompt”，而是当大模型已经足够强、能够长时间自主执行复杂任务时，工程重点如何从 prompt/context engineer 转向 **harness engineer**：为智能体设计可持续运行的环境、工作流、上下文检索机制、验证回路和通用工具集。

这篇内容的核心价值在于：它系统解释了为什么 2025 年底后 AI 编程范式发生变化，以及构建生产级自治 agent 时，真正的瓶颈不再只是模型能力，而是 **环境设计、上下文组织、验证机制和工作流编排**。这对于理解 coding agent、长期运行 agent、垂直领域 agent 和 agent-native 基础设施都很关键。

---

## 核心要点

- **2025 年底之后，AI 编程的拐点不只是模型更强，而是模型开始能稳定支撑长周期自治任务**：从简单 co-pilot 与短 loop，转向可连续推进复杂工作的长期 agent。
- **Harness engineer 是 context engineer 的升级版**：它关注的不只是单次会话里的 prompt 优化，而是多 session、多 agent、长任务链条下的环境、记忆、工具和工作流设计。
- **长周期 agent 成败的关键在“可读环境”**：如果代码、文档、计划、进度、设计决策、外部信息不在 agent 可访问环境里，对 agent 来说它们就等于不存在。
- **验证能力比生成能力更决定生产可用性**：让 agent 真正做成事，依赖端到端测试、浏览器自动化、结构化进度文件、git 记录、lint/架构检查等反馈回路，而不是口头要求“记得测试”。
- **通用工具往往比定制工具更有效**：模型通常更擅长 read/write/edit 文件、batch shell、git、浏览器调试这类训练分布里大量存在的原生工具，而不一定擅长复杂 bespoke JSON 工具接口。
- **自治 agent 的机会在垂直工作流而不只是通用聊天**：未来 6-12 个月的重要创业方向，是为某个高价值垂直场景构建像 OpenClaw 那样的常驻自治 agent。
- **智能体工程的核心不是“让模型更聪明”，而是“把系统设计成让模型足够聪明地工作”**。

## 正文

### 一、为什么说 2025 年底是 AI 编程范式的分水岭

视频开头的判断是：真正大的变化发生在 2025 年 12 月前后，但很多人并没有立刻意识到。变化不只是模型 benchmark 提升，而是模型与工具组合后，开始能够承接**长时间、自主推进、复杂且跨步骤的任务**。

作者用几个行业信号来说明这个拐点：

- 简单的 rough loop / for-loop agent 开始展现比过去明显更强的完成度；
- Cursor 展示了让模型大规模自主构建复杂项目的实验；
- Anthropic 展示多 agent 长时间协作完成编译器类项目；
- OpenClaw 这类“常驻、主动、长期运行”的 agent 系统迅速引起关注。

核心结论是：**问题正在从“模型能不能写一段代码”转向“如何让模型在长流程里持续有效工作”**。

### 二、什么是 Harness Engineer

作者把 harness engineer 看作 prompt engineer / context engineer 的进一步演化。

两者区别可以概括为：

| 角色 | 主要关注点 | 典型任务 |
|------|------------|----------|
| Prompt / Context Engineer | 单次会话里如何组织提示词与上下文 | 让模型在一个 context window 内完成更高质量输出 |
| Harness Engineer | 长周期、多 session、多 agent 系统如何持续运转 | 设计环境、工作流、记忆检索、验证回路、工具集与自动触发机制 |

也就是说，harness engineer 关心的是：

- 如何把长任务拆成 agent 能逐步推进的结构；
- 如何在不同 session 间保留有效状态；
- 如何让新 agent 接手时快速理解当前局面；
- 如何给 agent 适合的环境与工具；
- 如何确保 agent 真的验证过结果，而不是“以为自己完成了”。

### 三、从 AutoGPT 到 OpenClaw：自治智能体为什么现在才开始像样

视频回顾了 2023 年 AutoGPT 一类系统的局限：当时大家已经梦想让 AI 24/7 自治完成任务，但系统经常陷入低质量循环、目标漂移和错误累积，根本原因是模型与环境都还不成熟。

而 2025 年底后的变化是：

1. **模型的长程一致性更强**，能处理更长、更复杂的任务链；
2. **工具调用更稳**，模型在环境中行动的能力增强；
3. **工程系统开始补足“自治运行的支架”**，例如记忆层、定时触发、上下文检索、全电脑访问、消息通知等。

OpenClaw 在这里代表的不是“又一个桌面助手”，而是一类新的系统：

- 常驻运行；
- 可主动触发；
- 拥有本地环境访问能力；
- 有记忆与任务调度层；
- 能跨多个阶段持续推进复杂工作。

这标志着 agent 从“等人每一步发指令”的工具，走向“持续运作的工作系统”。

### 四、垂直领域机会：复制 OpenClaw，但不是做通用版

作者认为，未来 6-12 个月的重要机会，不是再做一个泛用聊天 agent，而是：

> 为某个垂直工作流构建一套长期运行、环境正确、工具正确、能闭环执行的自治 agent。

这里的关键不是模型本身，而是你是否真正理解该垂直场景的端到端流程，例如：

- 邮件营销
- 客服运营
- 销售线索处理
- 研发交付
- 数据分析与报告生成
- 内容生产与发布

如果你清楚一个领域的真实工作流、输入输出、常见异常、验证标准和外部依赖，就更有可能把 agent 做成生产系统，而不是 demo。

### 五、长周期 agent 的第一原则：环境必须“对 agent 可读”

视频重点引用了多个团队的经验：**真正的瓶颈不是 agent 不会写代码，而是 agent 每次进入新 session 都“不知道现在是什么情况”。**

因此，必须把环境做成对 agent 友好的“系统记录层”。

#### 1. 初始化 agent 负责建立环境

一个实践是先使用 initializer agent，而不是直接让 coding agent 上来就写功能。initializer agent 先做这些事：

- 启动或说明如何启动 dev server；
- 写出环境初始化脚本；
- 建立 feature list / task list；
- 创建 progress 文件记录已有进展；
- 留下清晰的初始 git 状态。

这样后续的 coding agent 每次接手时，就不是从混乱仓库里盲猜，而是从结构化环境里读取上下文。

#### 2. 用 feature list 防止“一次性乱做完”

视频提到一个做法：把项目拆成大量 feature/task，并为每个任务定义明确状态，默认都为 fail / 未完成。这样 agent 会：

- 先理解总体目标；
- 选最高优先级任务推进；
- 不容易因为做了局部功能就误判“整个项目完成”。

这相当于给 agent 提供了一个面向长期推进的计划系统。

#### 3. 用 progress 文件和 git 留下接力线索

每次 session 结束时，agent 不只是改代码，还要：

- 写清自己做了什么；
- 记录当前进展；
- 说明遗留问题；
- 提交 git 变更并写描述性 commit message。

这样下一次新的 context window 或新的 agent 进来时，可以快速重建局面。

### 六、验证回路比“再多提示几句”更重要

视频反复强调一个问题：模型天然倾向于高估完成度。你如果只在提示词里说“记得测试”，它通常不够。

真正有效的是给它**可操作、可反馈、可重复的验证机制**。

#### 1. 仅有单元测试/接口测试不够

在真实前端或全栈应用里，agent 可能通过单元测试，但功能仍然无法端到端正常工作。因为很多问题只会在：

- 浏览器交互
- DOM 状态
- 页面跳转
- 表单提交流程
- 截图可视结果

这些层面暴露。

#### 2. 浏览器自动化是自治 coding agent 的关键工具

作者特别提到，当给 agent 配置 Puppeteer MCP 或 Chrome DevTools 这类工具后，效果显著改善。原因是 agent 可以：

- 亲自启动应用；
- 重现 bug；
- 看 DOM / 页面状态；
- 截图比对；
- 验证修复是否真的生效。

这让 agent 从“读代码猜问题”升级为“在真实运行环境里观察—修改—验证”。

#### 3. 结构检查与架构约束应程序化

另一个经验是：不要把架构一致性、边界约束完全留给 agent 自觉维护，而要用系统规则强制执行，例如：

- lint
- structural tests
- domain boundary checks
- pre-commit hooks
- 自定义校验脚本

在传统团队里，这类约束往往是规模大了才做；但在 agent-heavy 开发里，它们应该更早出现，因为 agent 的代码生成速度很快，漂移也会很快。

### 七、文档不是可有可无，而是 agent 的“可检索工作记忆”

视频里有一个很重要的判断：

> 如果某些信息不在 agent 能访问到的环境里，那对 agent 来说它就等于不存在。

这意味着文档体系本身要重新设计，不再只是给人类读，而是要给 agent 检索和引用。

#### 1. AGENTS.md 更像目录，不该塞成巨型总文档

一个失败经验是把所有规则都堆进一个超大的 `AGENTS.md`，结果 agent 既读不完，也维护不好。

更有效的做法是：

- 把 `AGENTS.md` 当成目录 / TOC；
- 把架构、设计、执行计划、数据库、产品规格、安全约束、前端方案等拆到独立文档；
- 让 agent 按需检索相关文件。

这本质上是在做 **progressive disclosure**：不把所有信息一股脑塞进上下文，而是让系统支持按需取回。

#### 2. 把碎片化外部信息拉回仓库本地

视频还提到更进一步的做法：不仅把代码知识放进仓库，还把 Google Docs、Slack 消息、讨论记录等外部信息同步为本地可版本化工件。因为如果重要决策只存在于某个聊天工具里，而 agent 访问不到，它就无法纳入推理与执行。

对 agent 来说，一个可工作的仓库不只是代码库，而是**上下文可达的知识环境**。

### 八、通用工具优于复杂定制工具

视频借 Vercel 的 text-to-SQL 经验说明了一个反直觉结论：许多团队花大量时间做专用工具、复杂 prompt 和复杂 JSON tool schema，但效果反而脆弱、慢、难维护。

后来他们把系统简化到更少、更通用的工具后，性能和成功率反而更好。

作者总结这背后的原因大致是：

- 模型在训练中见过海量 shell、代码、文件操作模式；
- 对 read/write/edit/run 这类原生抽象更熟；
- 对 bespoke tool-calling schema 并不天然更强；
- 上下文清晰时，通用工具足够覆盖大量任务。

OpenClaw 的例子也类似：它的基础工具并不花哨，主要是：

- 读文件
- 写文件
- 编辑文件
- 执行 batch 命令
- 发消息

真正让系统强的，不只是工具数量，而是：

- 环境是否清晰；
- 文档是否可检索；
- 技能库是否可扩展；
- 工作流是否能持续闭环。

### 九、代码库要为 agent 重新设计，而不是默认沿用人类工程师习惯

视频中的另一个重点是：很多过去“等团队大了再做”的工程治理手段，在 agent 时代需要前置。

例如：

- 明确的领域边界和跨层约束；
- 更清晰的项目结构；
- 能一键启动的开发环境；
- worktree 级别可并行运行的 app；
- 便于自动化验证的测试与调试设施；
- 能留下连续痕迹的计划与进度文件。

换句话说，**不是把 agent 塞进旧代码库，而是把代码库改造成 agent 可操作环境。**

### 十、Harness engineer 的三条实践原则

结合视频中的多个案例，可以把 harness engineer 的核心方法浓缩成三条：

#### 原则 1：设计“可读环境”而不只是 prompt

重点不是一句神 prompt，而是让 agent 能迅速读懂：

- 当前目标
- 已完成进度
- 剩余任务
- 环境启动方式
- 设计边界
- 验证方式

#### 原则 2：建立“验证优先”的反馈闭环

给 agent 运行、观测、复现、截图、测试、提交、对比的能力，比多加几条行为要求有效得多。

#### 原则 3：优先使用模型原生熟悉的通用工具

在上下文系统足够好的前提下，优先使用通用读写编辑执行工具，再决定是否需要做高成本定制工具。

## 可复用方法 / 行动清单

### 为长期运行 coding agent 设计工作系统

1. **先做 initializer，再做 coding agent**
   - 先建立环境说明、启动脚本、任务列表、进度文件。
2. **把大目标拆成结构化 feature list**
   - 每个任务有描述、优先级、状态，默认未完成。
3. **让每次 session 都留下可接力痕迹**
   - 写 progress summary、保留 git commit、标注遗留问题。
4. **为 agent 提供真实验证能力**
   - dev server、端到端测试、浏览器自动化、截图/DOM 观察。
5. **把文档体系变成检索式知识环境**
   - `AGENTS.md` 做目录，架构/设计/安全/计划拆分成独立文档。
6. **程序化约束代码质量和架构边界**
   - pre-commit、lint、structural tests、boundary checks。
7. **先用通用工具，再决定是否专门造轮子**
   - read/write/edit/batch command/git/browser 往往已经足够强。

### 用于评估一个 agent 系统是否已进入“生产可用区间”

可以用下面的检查项：

- agent 是否能在新 session 中 5 分钟内恢复现场？
- 是否存在清晰的任务列表与状态定义？
- 是否可以自动启动和验证环境？
- 是否能端到端复现 bug 而不只是读代码？
- 是否能留下结构化进度记录与 commit 轨迹？
- 关键文档是否在本地可检索，而不是散落在外部工具里？
- 工具接口是否尽量简单、通用、稳定？

## 关键概念

| 概念 | 解释 |
|------|------|
| Harness Engineer | 面向长周期、多 session、多 agent 系统，设计环境、记忆、工具、验证与工作流的工程角色。 |
| Context Engineer | 关注单轮或单 session 中如何组织提示词和上下文，让模型在当前窗口里表现更好。 |
| Legible Environment | 对 agent 可读、可检索、可恢复状态的工作环境，包括代码、文档、计划、进度与工具。 |
| Initializer Agent | 在正式 coding 之前先建立环境、脚本、任务分解与状态记录的初始化智能体。 |
| Progress File | 用来记录当前进展、已完成事项和遗留问题的结构化文本，帮助下一个 session 快速接手。 |
| Verification Loop | 通过测试、浏览器自动化、截图、DOM 检查、git 提交等反馈机制，让 agent 真正验证结果。 |
| Generic Tools | 模型原生更熟悉的通用工具，如文件读写编辑、shell 命令、git、浏览器调试等。 |
| Progressive Disclosure | 不把所有信息塞进一个总文档，而是通过目录和分层文档实现按需检索。 |
| Agent-native Infrastructure | 为 agent 而不是只为人设计的文档、接口、环境和执行系统。 |

## 我的理解

这篇内容的价值在于，它把很多人对“AI 编程变强了”的模糊感受，具体化成了一套工程问题：**真正的竞争力不只是会不会用模型，而是能不能把仓库、文档、工具、验证和工作流改造成 agent 能长期工作的环境。**

如果说过去的重点是“写好 prompt”，那接下来的重点更像“搭好工地”。模型只是工人，harness 才是决定这群工人能否连续、稳定、高质量产出的施工系统。
