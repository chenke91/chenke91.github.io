# 从 Vibe Coding 到 Agentic Engineering：AI 编程的新范式

## 简介

本文整理自 Sequoia Capital 的访谈视频 [Andrej Karpathy: From Vibe Coding to Agentic Engineering](https://youtu.be/96jN2OCOfLs)。Andrej Karpathy 在对话中回顾了从“vibe coding”到“agentic engineering”的变化：AI 编程工具不再只是补全代码，而是在可验证任务、软件安装、知识重组、部署流程等场景中，逐渐成为新的计算与工程范式。

这篇内容的核心价值在于：它不是在讨论“AI 能不能写代码”，而是在重新定义**人类工程师与智能体的分工**。Karpathy 认为，vibe coding 提高了所有人的软件能力下限，而 agentic engineering 则是在专业软件工程中保持质量、安全、设计和可验证性的前提下，大幅提高上限。

---

## 核心要点

- **2025 年底前后，AI 编程工具发生了质变**：Karpathy 观察到最新模型生成的代码块越来越少需要修正，这让他重新开始大量编程，并感觉“作为程序员从未如此落后”。
- **Software 3.0 的核心是用上下文和提示编程**：Software 1.0 是显式代码，Software 2.0 是数据集和神经网络权重，Software 3.0 则是通过 context window 操控 LLM 这个“解释器”。
- **新范式不是把旧软件做快，而是让过去不存在的东西变得可能**：例如用 LLM 把一堆文档重新编译成知识库，或者直接让模型处理菜单图片并生成增强后的视觉结果。
- **LLM 最容易自动化“可验证”的领域**：传统计算机自动化可明确编码的任务，LLM 更擅长自动化能被验证、能给 reward 的任务，如数学、代码、测试、部分结构化判断。
- **Vibe coding 和 agentic engineering 不是一回事**：前者提高软件创作门槛的下限，后者关注专业工程质量、安全性、可靠性和系统设计。
- **未来工程师更重要的是品味、判断、规格设计和监督**：API 细节可以交给智能体，但系统边界、用户 ID、抽象设计、安全约束等仍需人类负责。
- **“可以外包思考，但不能外包理解”**：智能体能处理大量信息，但人仍要理解要构建什么、为什么值得做，以及如何指挥智能体。

## 正文

### 一、AI 编程工具的质变：从辅助补全到可信任代码块

Karpathy 说，过去一年他一直在使用 Claude Code、Codex 等 agentic coding 工具。早期这些工具“有帮助”，但经常需要人工修正。真正的变化出现在 2025 年底左右：

- 最新模型生成的代码块开始“基本可用”；
- 他越来越少纠正模型输出；
- 信任逐步增加；
- 自己重新进入高强度编程状态；
- side project 文件夹迅速膨胀。

这也是他为什么说自己“从未如此落后”：不是因为不会编程，而是工具能力迭代太快，专业程序员需要重新学习新的协作方式。

### 二、Software 3.0：上下文窗口成为新的编程接口

Karpathy 继续沿用自己过去的分层说法：

| 阶段 | 编程方式 | 核心对象 |
|------|----------|----------|
| Software 1.0 | 写显式代码 | 规则、函数、程序 |
| Software 2.0 | 组织数据集和训练神经网络 | 数据、目标函数、权重 |
| Software 3.0 | 编写提示词和上下文 | prompt、context window、LLM |

在 Software 3.0 中，LLM 像一个新的解释器。开发者真正操控的是：

- 上下文窗口里放什么；
- 给模型什么任务描述；
- 提供哪些环境信息；
- 如何让模型在循环中观察、行动、调试。

这意味着“程序”不再一定是传统代码，也可能是一段复制给 agent 的说明文本。

### 三、Agent 作为新的软件安装器

Karpathy 用 OpenClaw 的安装方式举例。传统安装脚本通常要兼容各种平台和环境，因此 shell script 会不断膨胀，变得复杂脆弱。

而 Software 3.0 的安装方式可能是：

> 复制一段说明文本，交给 agent，让它根据当前电脑环境完成安装。

这种方式的优势是：

- 不需要把所有平台差异写死在脚本里；
- agent 可以观察当前环境；
- agent 可以动态调试；
- 安装流程可以变成“智能动作序列”，而不是静态脚本。

这说明 agent 正在成为一种新的“软件安装器”或“操作执行层”。

### 四、从 MenuGen 看新范式：有些应用本身可能不再需要存在

Karpathy 提到自己做过一个 MenuGen 项目：用户拍餐厅菜单，系统 OCR 菜单项，再用图像生成模型生成菜品图片并重新展示菜单。

在旧范式下，这需要一个完整 app：

1. 上传菜单图片；
2. OCR 菜单文字；
3. 解析菜名；
4. 调用图像生成模型；
5. 重新渲染菜单界面。

但 Software 3.0 的版本可能极其简单：

> 把菜单照片交给 Gemini，并要求它用图像模型直接在菜单上叠加菜品图片。

这个例子让 Karpathy 意识到：很多旧范式中的应用可能只是为了弥补模型能力不足而存在。随着模型能直接处理图片、文本和视觉输出，中间层软件可能会被压缩甚至消失。

### 五、新机会不是“把旧流程加速”，而是创造过去不可能的工作流

Karpathy 特别强调，不要只问：

> 以前做的事情，现在能不能更快？

更重要的问题是：

> 以前不存在的事情，现在是否变得可能？

他举了知识库的例子：过去没有一段传统代码可以自动把大量事实、文档和材料重新组织成个人或组织知识库；但 LLM 可以把原始资料“重新编译”为新的结构。

这和本知识库项目高度相关：raw 文件不是简单归档，而可以通过 LLM 被重新排列、提炼、分类、生成 wiki，成为新的理解界面。

### 六、可验证性：LLM 自动化能力的关键边界

Karpathy 认为，传统计算机擅长自动化**可明确编码**的事情，而当前 LLM 更擅长自动化**可验证**的事情。

原因在于前沿模型训练大量依赖强化学习环境：

- 如果任务结果可以验证，就能给模型 reward；
- 如果能给 reward，模型能力就会快速提升；
- 因此数学、代码、测试等领域会出现能力尖峰；
- 不可验证或难验证领域则表现更“锯齿状”。

这解释了所谓 jagged intelligence：模型可以重构十万行代码、发现安全漏洞，却可能在“车洗店距离 50 米，应该开车还是走路”这种常识问题上给出荒唐答案。

### 七、创业和产品机会：寻找可验证但尚未被大模型实验室充分覆盖的领域

对创业者来说，Karpathy 的建议可以概括为：

1. 找到一个有价值的问题；
2. 判断它是否可以构造验证环境；
3. 如果可以验证，就有机会通过 fine-tuning、RL 环境或高质量数据获得优势；
4. 不要只盯着大模型实验室已经重点优化的数学和代码；
5. 寻找“经济价值高、可验证、但还没进入主流训练分布”的领域。

他没有直接透露具体领域，但强调：可验证性本身就是当前范式下的技术杠杆。

### 八、Vibe Coding vs Agentic Engineering

Karpathy 对两者做了非常清晰的区分：

| 概念 | 作用 | 关注点 |
|------|------|--------|
| Vibe Coding | 提高所有人的软件创作下限 | 让非专业或普通用户也能做出软件 |
| Agentic Engineering | 提高专业工程上限 | 在保持质量、安全、可靠性的前提下加速工程 |

Vibe coding 可以让每个人都“凭感觉”做出东西，这是巨大的普惠能力。但专业软件不能只靠感觉：

- 不能引入安全漏洞；
- 不能放弃代码质量；
- 不能逃避系统设计责任；
- 不能因为 agent 能写代码就忽视验证。

Agentic engineering 的核心问题是：

> 如何协调一组强大但随机、会犯错的 agent，在不牺牲质量标准的前提下更快地构建系统？

### 九、AI Native 工程师的差异：工具利用、环境投资和大项目能力

Karpathy 认为，AI native 工程师和普通使用者的差异，不只是会不会打开 Claude Code 或 Codex，而是：

- 是否充分利用工具特性；
- 是否持续投资自己的开发环境；
- 是否会组织 agent 完成大项目；
- 是否能设计大型任务、验证结果、保持质量；
- 是否能把 agent 当作工程系统的一部分。

他还指出，招聘流程也应该变化。传统算法题并不能很好评估 agentic engineer。更合理的测试可能是：

- 给候选人一个大项目，例如实现一个 Twitter clone；
- 要求它足够好、足够安全；
- 再用多个 agent 尝试攻击或破坏系统；
- 看候选人如何设计、实现、验证和防守。

### 十、人类仍然负责品味、判断、规格和监督

当 agent 能写越来越多代码，人类更重要的能力不是记住 API 细节，而是：

- 系统设计；
- 规格定义；
- 工程判断；
- 审美和品味；
- 安全意识；
- 监督和验收；
- 判断什么值得做。

Karpathy 举了 MenuGen 中的一个错误：agent 曾试图用 Stripe 邮箱和 Google 邮箱来关联用户资金。但这两个邮箱可能完全不同，用邮箱做资金归属是错误设计，应该使用稳定的 user ID。

这类问题不是 API 细节，而是系统建模与工程判断。agent 可以填补实现细节，但人必须负责关键抽象和约束。

他也提到，自己已经不太记得 PyTorch、NumPy、Pandas 里某些 API 参数到底叫 `dim`、`axis`、`keepdims` 还是别的名字，因为这些细节可以交给 agent。但他仍需要理解 tensor、view、storage、copy 等底层概念，否则会做出低效或错误设计。

### 十一、不要把 LLM 当动物，而要理解其“幽灵式智能”

Karpathy 用“不是动物，而是幽灵”来描述 LLM：它们不是由进化塑造出来、有内在动机、好奇心或生存目标的动物智能，而是由预训练数据和 reward function 塑造出来的统计模拟系统。

这个比喻的意义在于提醒使用者：

- 对模型发脾气没有意义；
- 它不会因为像动物一样被训练而真正“理解”奖惩；
- 它的能力分布由数据和 RL 环境塑造；
- 它可能在某些回路里极强，在另一些回路里极弱；
- 使用者要保持怀疑，并持续探索边界。

### 十二、Agent-native 世界：基础设施要为智能体重写

Karpathy 认为，今天大多数软件和文档仍然是为人类写的。但如果 agent 真的成为执行主体，很多基础设施都要变成 agent-native。

他吐槽现在的文档仍然经常告诉用户“去这个页面、点那个按钮、配置这个设置”。但在 agent 时代，更好的文档应该告诉用户：

> 复制这段内容给你的 agent。

未来的基础设施应该更适合 agent：

- 有清晰的机器可读接口；
- 有稳定的数据结构；
- 有传感器和执行器；
- 能让 agent 直接部署、配置、接入服务；
- 减少人类在网页菜单中手动点击。

MenuGen 的部署过程就是反例：写代码本身可能很快，但部署、DNS、第三方服务设置、后台配置仍然需要人频繁介入。这些环节未来都应该被 agent-native 化。

### 十三、教育与理解：不能外包理解

访谈最后，Karpathy 提到一句让他反复思考的话：

> You can outsource your thinking, but you can't outsource your understanding.

可以把思考过程、信息处理、代码实现交给 agent，但理解仍然要进入人的大脑。因为人仍然要决定：

- 我们要构建什么？
- 为什么值得构建？
- 应该如何指挥 agent？
- 如何判断输出是否正确？
- 如何把信息转化为自己的认知？

这也是他对 LLM knowledge bases 感兴趣的原因：把文章、资料、事实重组为 wiki，本质上是在帮助人从不同投影角度理解同一批信息。

## 可复用方法 / 行动清单

### 面向个人开发者

- 不要只把 AI 编程工具当作 autocomplete，要把它当作可以观察环境、执行动作、调试问题的 agent。
- 为自己的 agentic coding 工作流投资环境：提示词、脚本、工具、测试、验证、部署流程。
- 把注意力从 API 细节转移到系统设计、规格定义、质量控制和验收。
- 对 agent 输出保持怀疑，尤其是身份、权限、资金、数据归属、安全边界等关键抽象。
- 用测试、lint、类型检查、审查、攻击模拟等方式增加任务可验证性。

### 面向创业者 / 产品设计者

- 寻找“可验证但尚未被充分自动化”的高价值领域。
- 不只思考如何让旧流程更快，也要思考过去不存在的新工作流。
- 设计 agent-native 产品时，不要只给人类 UI，也要提供 agent 可读、可操作、可验证的接口。
- 重新审视哪些 app 只是旧范式下的中间层，是否可以被模型直接处理输入输出替代。

### 面向知识管理

- raw 资料不应只是保存，而应通过 LLM 重新编译为 wiki、索引、框架和问答入口。
- 一篇材料可以被多次投影：摘要、概念图、行动清单、问题列表、反例、关联笔记。
- 知识库的目标不是外包理解，而是增强理解。

## 关键概念

| 概念 | 解释 |
|------|------|
| Vibe Coding | 用 AI 低门槛快速构建软件，提升所有人的软件创作下限。 |
| Agentic Engineering | 将 agent 纳入专业工程流程，在保持质量、安全和可靠性的前提下加速开发。 |
| Software 3.0 | 以 prompt 和 context window 为主要接口，通过 LLM 进行信息处理和执行的新软件范式。 |
| Verifiability | 任务结果是否容易被验证。越可验证，越容易通过 RL、测试、评分器提升模型能力。 |
| Jagged Intelligence | LLM 能力分布不平滑：某些高难任务极强，某些常识任务却很弱。 |
| Agent-native Infrastructure | 为 agent 而不是只为人类设计的基础设施、文档、接口和工作流。 |
| Understanding | 人类对目标、系统、约束和意义的内化理解，不能简单外包给模型。 |

## 我的理解

这篇访谈对个人知识库特别有价值，因为它解释了为什么“把 raw 整理成 wiki”本身就是 Software 3.0 的典型用法：LLM 不只是帮助摘要，而是在把原始信息重新编译成更适合理解、复用和行动的结构。

对工程实践来说，最重要的启发是：未来优秀工程师不是被 agent 替代，而是更像 agent 的导演、规格设计者和质量负责人。细节实现会越来越便宜，但理解、判断、验证和品味会更值钱。
